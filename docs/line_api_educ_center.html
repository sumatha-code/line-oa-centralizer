Phase 1: Multi-Account Gateway 4 Containers (Docker Compose)

🎯 เป้าหมายของระบบ

ทลายข้อจำกัดของ LINE OA ที่กำหนด Webhook ได้เพียง 1 URL โดยสร้าง Gateway กลางแบบ Multi-Account เพื่อรับ Webhook จาก LINE OA หลายๆ บัญชีพร้อมกัน ผ่าน Dynamic Route จากนั้นทำการจัดคิว (Queue) ลง Redis และส่งต่อ (Forward) ไปยัง Chatbot หรือระบบปลายทางของแต่ละบัญชีตามที่ตั้งค่าไว้ พร้อมระงับ/อนุมัติสิทธิ์การใช้งาน API (API Key) และจัดสรรสิทธิ์ผู้ดูแลระบบย่อย (Admin Whitelist) แยกรายบัญชีได้

🛠️ Tech Stack & Services

  • Web Service: Next.js (TypeScript)
  • Queue Worker: Node/TS Background Service
  • Database: PostgreSQL + Drizzle ORM
  • Queue/Broker: Redis
  • Auth: Google Login (Auth.js)

⚖️ การตัดสินใจทางเทคนิค (Decisions)

  • 1. Multi-account LINE Support จัดเก็บรายละเอียดบัญชี LINE OA (ID, Secret, Tokens) ไว้ใน Database เพื่อรองรับบอทหลายตัวพร้อมกันโดยไม่มีขีดจำกัด
  • 2. Hybrid Admin Whitelist (Fine-grained) มี Super Admin คุมระบบ และสามารถเชิญผู้ดูแลระบบย่อยในตาราง whitelist พร้อมจำกัดสิทธิ์ให้เข้าถึงหน้าจอจัดการได้เฉพาะบางบัญชี LINE OA เท่านั้น
  • 3. Redis Queue & Separate Worker Process เลี่ยงข้อจำกัด Webhook 2 วินาที ด้วยการเก็บลงคิวทันที และย้ายตัววิเคราะห์ประมวลผลไปทำใน Worker Container แยกเพื่อความนิ่งสูงสุด
  • 4. API Key Access Scoping แอปย่อยที่เรียกใช้ช่องทางส่งข้อความส่วนกลาง จะต้องแนบ API Key ที่ได้รับสิทธิ์เชื่อมโยงกับ LINE Account นั้นๆ เท่านั้น ป้องกันการสวมรอยส่งข้อความข้ามระบบ

🗄️ โครงสร้างฐานข้อมูลหลัก

line_accounts
เก็บ credentials บัญชี LINE OA และ URL ส่งต่อบอทย่อย
admin_whitelist
เก็บอีเมลแอดมินย่อย และตารางสิทธิ์เชื่อมกับ line_accounts
api_keys
เก็บ Hashed API Key และสิทธิ์เชื่อมกับ line_accounts
usage_logs
ประวัติการใช้ API (ส่งหา LINE Account ไหน, สถานะใด)
users / groups
โปรไฟล์ผู้ใช้และกลุ่ม LINE ที่เข้ามาใช้งานบอทของระบบ
webhook_events
เก็บ ID Webhook ป้องกันการทำงานซ้ำซ้อนจาก LINE
