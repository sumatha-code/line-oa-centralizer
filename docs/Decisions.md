# Technical Decisions (ADR)
บันทึกเหตุผลการตัดสินใจทางเทคนิค ห้ามเปลี่ยนสถาปัตยกรรมเหล่านี้โดยพลการ

1.  **ใช้ PostgreSQL Image ปกติแทน Supabase Self-hosted:** * *เหตุผล:* ต้องการให้ระบบมีความเบา บริโภคทรัพยากรบน Proxmox น้อยที่สุด และดูแลจัดการง่ายด้วย Drizzle ORM
2.  **ติดตั้ง Redis เข้ามาตั้งแต่ Day 1:** * *เหตุผล:* ป้องกันปัญหา LINE Webhook Timeout (2 วินาที) อย่างเด็ดขาด โดยให้ API รับ Request แล้วโยนลง Redis ทันที การประมวลผลที่ใช้เวลาจะทำใน Worker/Consumer
3.  **ใช้ Google Login สำหรับ Admin Panel:** * *เหตุผล:* ไม่ต้องการสร้างระบบจัดการรหัสผ่านผู้ดูแลระบบเอง ป้องกันปัญหาลืมรหัสผ่าน และจำกัดสิทธิ์เฉพาะอีเมลขององค์กร (เช่น `@su.ac.th`) ได้ง่าย
4.  **ใช้ Ant Design (Dark Mode):** * *เหตุผล:* มี Data Table และ Form Component ที่ครบถ้วนและทรงพลัง เหมาะสำหรับทำระบบ Admin/Backoffice ที่สุด ลดเวลาในการสร้าง UI เอง
5.  **สถาปัตยกรรมข้อความแชท (Lightweight Router):** * *เหตุผล:* Gateway จะไม่เก็บประวัติข้อความแชท (Chat History) ลงฐานข้อมูลตัวเอง แต่จะทำหน้าที่ Forward Payload ไปให้ระบบ Chatbot จัดการต่อ เพื่อลดภาระระบบกลางและหลีกเลี่ยงข้อกังวลด้านข้อมูลส่วนบุคคล (PDPA)
6.  **ระบบ Routing Rules และ Forward Webhook แบบหลายปลายทาง (Multi-destination):** * *เหตุผล:* เพื่อรองรับกรณีที่บอทของแต่ละหลักสูตร/งานในหน่วยงานย่อยต้องการใช้งานร่วมกันภายใต้บัญชี LINE OA เดียวกัน ระบบจะดึงรายชื่อและจัดส่งต่อ (Forward) Payload Webhook ไปยังปลายทางบอททุกแห่งที่ผูกไว้ขนานกัน (Parallel Dispatching) ผ่าน Queue Worker โดยระบุผ่านโครงสร้างแบบฟอร์ม Dynamic List บนหน้าจอผู้ดูแลระบบ
7.  **การจำกัดสิทธิ์ผู้ดูแลระบบ (Hybrid Whitelist):** 
    * *เหตุผล:* ใช้ระบบสิทธิ์ 2 ระดับ คือ 1) **Super Admin** จำนวนหนึ่ง กำหนดตายตัวผ่าน Environment Variable (`SUPER_ADMIN_EMAILS`) เพื่อป้องกันระบบล็อกและเป็นผู้เริ่มต้นระบบ 2) **Whitelisted Admins** คือผู้ใช้ทั่วไปที่ Super Admin สามารถกดเพิ่มอีเมลเข้ามาในฐานข้อมูลตาราง `admin_whitelist` ได้ผ่านหน้า Admin Panel และสามารถจัดการสิทธิ์ในการเข้าถึงและดูแลเฉพาะบาง LINE Accounts ที่ระบุไว้ได้
8.  **การรัน Queue Consumer แบบแยก Container และขยายขนาดเป็น 4 Replicas:**
    * *เหตุผล:* เพื่อให้การรันคิวไม่ส่งผลกระทบต่อประสิทธิภาพเว็บหลัก และเพื่อรองรับการตอบกลับส่งต่อข้อมูลของบอทพร้อมกัน 3-4 บัญชีโดยไม่มีความล่าช้า จึงตกลงรัน Queue Worker แยกเป็นคอนเทนเนอร์เฉพาะ และตั้งค่าขยายขนาดคอนเทนเนอร์เป็น 4 Replicas (`replicas: 4`) ใน Docker Compose ให้ดึงงานจาก Redis มาแบ่งงานกันทำโดยอัตโนมัติ (Queue Load Balancing)
9.  **การจัดการสิทธิ์เข้าถึงบัญชี LINE (Multi-account Access Control):**
    * *เหตุผล:* เพื่อป้องกันข้อผิดพลาดในการส่งข้อความและการดูข้อมูลข้ามแผนก/ระบบย่อย ให้สิทธิ์สำหรับ API Key และผู้ดูแลระบบ (Admins) ถูกจำกัดและตรวจสอบแยกตามรายบัญชี LINE Account (ตาราง N-to-M mapping) โดยมี Super Admin เท่านั้นที่สามารถแก้ไขความสัมพันธ์เหล่านี้ได้
10: **การทำงานหลัง Cloudflare Reverse Proxy และความปลอดภัยในการจัดเก็บคีย์:**
    * *เหตุผล:* การเปิดบริการโดเมนสาธารณะ (`https://line-api.educ-su.work`) ผ่าน Cloudflared Tunnel ส่งผลให้ตัวแปรเบื้องหลังระบบล็อกอิน Google OAuth คลาดเคลื่อน (Redirect URI mismatched) จึงใช้การเปิดใช้งาน `TRUST_HOST=true` เพื่อให้ระบบเบื้องหลัง (NextAuth) อ่านและไว้วางใจ HTTP Headers ที่ Proxy ส่งมาได้อย่างถูกต้อง และได้มีการแยกความลับ (Secrets) ออกจากโค้ดและเอกสารแบบ 100% โดยบันทึกเฉพาะในไฟล์ `.env` ที่ปลอดภัยบนเครื่องเซิร์ฟเวอร์เท่านั้น
