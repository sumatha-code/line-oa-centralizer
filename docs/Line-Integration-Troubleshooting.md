# คู่มือวิเคราะห์ข้อผิดพลาดและวิธีแก้ไขสำหรับการเชื่อมต่อ LINE API
คลังความรู้สะสมปัญหาและการออกแบบทางเทคนิคเพื่อป้องกันปัญหาซ้ำซ้อน (LINE Troubleshoots & Gotchas)

---

## 1. ปัญหา Webhook Timeout (การตอบกลับช้าเกิน 2 วินาที)
* **อาการ:** ระบบทำงานสำเร็จ แต่ LINE ส่ง Webhook ซ้ำเข้ามาเรื่อยๆ ส่งผลให้มีการแจ้งเตือนซ้ำหรืออนุมัติซ้ำ
* **สาเหตุ:** LINE Messaging API กำหนดว่า Endpoint ปลายทางต้องตอบกลับสถานะ HTTP Status Code `2xx` **ภายใน 2 วินาที** หากตัวแอปพลิเคชันทำการวิเคราะห์ข้อมูล ส่งเมล์ หรือยิง API ต่อทันทีใน Thread หลักจนหน่วงเกิน 2 วินาที LINE จะถือว่าส่งไม่สำเร็จและพยายามยิงซ้ำ (Redelivery)
* **วิธีแก้ไข:** 
  * ทันทีที่ Endpoint ได้รับ Webhook ให้ตรวจสอบความปลอดภัยเบื้องต้น (เช่น Signature) และบันทึก Event ลงฐานข้อมูลแบบด่วน จากนั้นตอบกลับ `200 OK` ทันทีใน Thread หลัก
  * แยกการทำงานที่กินเวลานาน (เช่น การสร้างข้อความ Flex, การยิง Push Message, หรือการอัปเดตสถานะแบบ Async) ไปทำงานเบื้องหลัง (Background Process/Async Task) ของระบบ หรือโยนลงระบบคิว (Message Queue)

---

## 2. ปัญหา Webhook Event Redelivery (ประมวลผลข้อความซ้ำซ้อน)
* **อาการ:** ผู้อนุมัติเห็นข้อความแจ้งเตือนเดิมขึ้นมาซ้ำๆ หรือระบบทำรายการซ้ำเนื่องจาก LINE ส่ง Event ซ้ำเข้ามาจริง
* **สาเหตุ:** 
  * LINE อาจส่งข้อความเดิมเข้ามาซ้ำเนื่องจากปัญหาเน็ตเวิร์กระหว่างทางสะดุด
  * **กรณีศึกษาบั๊กที่เคยเกิดขึ้น:** ตัวรับ Webhook ดึงชื่อฟิลด์ ID ไม่ตรงกับโครงสร้างข้อมูลจริงของ LINE (เช่น ไปดึงฟิลด์ `eventId` แทนที่จะเป็น `webhookEventId`) ส่งผลให้ตัวแปรมีค่าเป็น null/undefined และหลุดรอดจากระบบกรองซ้ำ (Deduplication) ข้ามการบันทึกข้อความจริงไปเงียบๆ โดยไม่ฟ้อง error
* **วิธีแก้ไข:**
  * ตรวจสอบว่าฟิลด์ที่สกัดออกมาจาก Request JSON คือ `webhookEventId` (ซึ่งเป็นฟิลด์มาตรฐานของ LINE สำหรับระบุ Webhook Event ID) เท่านั้น
  * ใช้ฟิลด์ `webhookEventId` นี้เป็น Primary Key ของตารางบันทึก Webhook
  * ก่อนเริ่มประมวลผล Webhook ทุกครั้ง ให้ทำ Query เช็คในตาราง `line_webhook_events` ก่อนว่ามีไอดีนี้อยู่แล้วหรือไม่ หากเคยมีอยู่แล้ว ให้ข้ามการประมวลผลและตอบกลับ `200 OK` ทันที

## 3. ปัญหา LINE ID Token Leakage & Spoofing (การสวมสิทธิ์ผู้อนุมัติ)
* **อาการ:** มีผู้ใช้บางคนสามารถส่งคำสั่งอนุมัติแทนผู้บริหารได้ หรือปลอมแปลงสิทธิ์การเข้าใช้งานหน้า LIFF
* **สาเหตุ:** ฝั่ง Client-side ของ LIFF (ในมือถือ) ส่งเฉพาะค่าสตริง `lineUserId` ไปหาหลังบ้าน ซึ่งผู้ไม่หวังดีสามารถดักจับเน็ตเวิร์กและเปลี่ยนไอดีเป็นไอดีของผู้บริหารเพื่อสวมรอยอนุมัติ
* **วิธีแก้ไข:**
  * **ห้ามส่งและห้ามเชื่อถือ** `lineUserId` ที่ส่งตรงจากฝั่ง Client เด็ดขาด
  * ให้ฝั่ง Client ส่งเฉพาะ ID Token ตัวจริงที่ได้จาก `liff.getIDToken()` เท่านั้น
  * ฝั่ง Server-side จะต้องนำ raw ID Token นี้ไปยิงตรวจสอบความถูกต้องกับ Endpoint ของ LINE (`POST https://api.line.me/oauth2/v2.1/verify`) โดยตรวจสอบ `client_id` ร่วมกับ `LINE_LOGIN_CHANNEL_ID` เพื่อยืนยันความปลอดภัย จากนั้นจึงดึงสิทธิ์ Claim `sub` ใน Token มาใช้แทน `lineUserId` ที่ปลอดภัย

## 4. ปัญหา การตรวจสอบ Signature Webhook ผิดพลาด (Signature Mismatch)
* **อาการ:** ตรวจสอบ Signature ไม่ผ่าน (ได้ 403 Forbidden) ทั้งที่เป็นข้อความจริงจาก LINE
* **สาเหตุ:** การตรวจสอบ Header `x-line-signature` ต้องนำ **Raw Request Body (Buffer/String ดั้งเดิม)** มาคำนวณร่วมกับ `LINE_CHANNEL_SECRET` การเผลอไปแปลง request body เป็น JSON object ก่อนทำ hashing จะทำให้ค่า Hash เปลี่ยนไปและตรวจสอบไม่ผ่าน
* **วิธีแก้ไข:**
  * ต้องดึงข้อมูลในรูปของ text/buffer ตรงๆ จาก HTTP request ดั้งเดิมก่อนการพาร์ส JSON
  * เขียนโค้ดคำนวณ HMAC-SHA256 และเปรียบเทียบในรูปของ Base64 string

---

## 5. ปัญหา User ID ไม่ตรงกันระหว่าง Login และ Messaging API (Provider Mismatch)
* **อาการ:** ผู้อนุมัติเข้าหน้าเว็บผ่าน LIFF ได้ แต่ระบบไม่สามารถค้นหาข้อมูลในตาราง `line_mappings` หรือไม่สามารถส่ง Push Message แจ้งเตือนกลับไปหาผู้ใช้ได้
* **สาเหตุ:** LINE Login Channel (ที่ใช้ใน LIFF) และ LINE Messaging API (ที่ใช้แจ้งเตือน/OA) ไม่ได้สร้างอยู่ภายใต้ **Provider เดียวกัน (Developer Console)** ส่งผลให้ `lineUserId` ของผู้ใช้คนเดียวกันมีค่าไม่ตรงกัน (LINE จะสุ่ม User ID แยกตาม Provider เพื่อความเป็นส่วนตัว)
* **วิธีแก้ไข:**
  * ตรวจสอบว่าทั้ง LINE Login Channel และ Messaging API Channel ถูกสร้างไว้ภายใต้ **Provider ชื่อเดียวกัน** เท่านั้น ห้ามข้าม Provider

---

## 6. ปัญหา Race Condition เมื่อกดปุ่ม LINE Flex Message พร้อมกัน
* **อาการ:** ผู้อนุมัติร่วมในกลุ่ม (เช่น ระดับหัวหน้างาน 3 คน) กดปุ่มอนุมัติเกือบพร้อมกันผ่าน Flex Message ในแชท ส่งผลให้เกิดการชนกันของสถานะและ Database Lock
* **สาเหตุ:** การเขียนอัปเดตสถานะแบบไม่มีการล็อกข้อมูล ส่งผลให้มีคนกดพร้อมกันแล้วเปลี่ยนค่าสถานะพร้อมกัน หรือเกิดสภาวะ Database lock ของ SQLite
* **วิธีแก้ไข:**
  * รันข้อมูลกระบวนการอัปเดตสถานะและการบันทึกทั้งหมดภายใต้ Database Transaction
  * กำหนดโครงสร้าง **Idempotency Key (คีย์ป้องกันซ้ำ)** เช่น บันทึกในตาราง `approval_actions` โดยมี Unique constraint ของคีย์ที่สร้างจากสูตร `requestId + approverId + action` หากมีรายการอื่นยิงมาซ้ำคีย์นี้จะติด Unique constraint และตัดระบบทันที
  * ตั้งค่าหน่วงเวลาฐานข้อมูล `busy_timeout` ให้ไม่น้อยกว่า 5000ms ใน SQLite เพื่อรอจังหวะคิวเขียน
